# metal/ — on-device (Metal + CoreML) inference

**Status: complete — descriptor + decode path verified on-device.** The GPU kernels
(`rdfHistogram` with full periodic-image enumeration + atomic-min nearest distance, and
`envSample`), the CPU post-processing, env 4-draw pooling, composition decode, and RAPS
conformal sets are all implemented and pass parity against the Python reference (see
_Verification_). Build/verify on a Mac (Metal + CoreML are Apple-only); the parity target
in `parity/golden.json` is generated by the Python path and is platform-independent.
The only item left is a perf optimization (uniform-grid neighbor search) — the brute-force
`O(N²·images)` path is correct, just not fast on million-atom files.

## The idea

The Python pipeline is `parse → descriptors → set-transformer → decode`. Only the
descriptor stage is expensive at scale, and it's plain histogram math — a perfect fit for
a GPU kernel. `InaccurateRDFCalculator` computes the per-type-pair descriptors on Metal
and hands them to the CoreML-exported network (`env_codfull_sharp30.mlpackage`, 3.6 MB
fp16; committed int8 ONNX is for the desktop lite path).

"Inaccurate" is a deliberate design license: chemistry is decided by bond-length **peaks**
and **coordination**, both robust to subsampling RDF centers or using a coarse neighbor
search. So the on-device descriptor may approximate `g_ab(r)` — parity is defined by
*predictions*, not bit-exact histograms.

## Layout

```
Package.swift
Sources/InaccurateRDFCalculator/
  Constants.swift          descriptor constants — MUST match descriptors.py
  Shaders.metal            GPU: rdfHistogram (image-enum + atomic-min), envSample
  DescriptorComputer.swift Metal dispatch + CPU normalize/CN/peak/frac/glob + env bucketing
  Helpers.swift            normalize, gaussian smear, percentile, box volume
  CoreMLRunner.swift       pad to T=8, build MLMultiArrays, run, slice log_probs
  Decode.swift             CompositionPrior (PMI + charge-neutrality beam) + RAPS conformal
  Oxidation.swift          per-Z oxidation states (port of decode._OX)
  TypeSHI.swift            end-to-end predict() -> ranked candidates + conformal set
  Resources/decode_assets.json  baked PMI matrix + conformal calibration (see tools/)
Sources/ParityCheck/main.swift   diff Metal descriptors/decode vs parity/golden.json
Sources/DescDump/main.swift      dump Metal descriptors for an arbitrary structure (JSON)
tools/gen_decode_assets.py       bake decode_assets.json from weights/*.npz
parity/                    golden reference + generators + verification scripts
```

The Shader is bundled as a resource and compiled at runtime via
`device.makeLibrary(source:)` — SwiftPM copies (doesn't compile) `.metal` files, and
runtime source-compilation is the portable fix that works on both macOS and iOS.

## Using it from an app

```swift
import InaccurateRDFCalculator

// env_codfull_sharp30.mlpackage ships in the GitHub Release tarball (fp16 CoreML).
let model = Bundle.main.url(forResource: "env_codfull_sharp30", withExtension: "mlpackage")!
let shi = try TypeSHI(modelURL: model)

let structure = Structure(
    positions: atoms.map { SIMD3<Float>($0.x, $0.y, $0.z) },  // Ångström, absolute
    typeIds: atoms.map { UInt32($0.typeId) },                 // 0-based, contiguous
    cell: (SIMD3(ax,ay,az), SIMD3(bx,by,bz), SIMD3(cx,cy,cz))) // nil for a molecule

let preds = try shi.predict(structure, topK: 5)      // [TypePrediction], one per type id
for (t, p) in preds.enumerated() {
    let top = p.candidates.map { "\($0.symbol) \(Int($0.prob * 100))%" }.joined(separator: ", ")
    let set = p.conformalSet.map(\.symbol).joined(separator: " ")
    print("type \(t): \(top)   [90% set: \(set)]")
}
```

`predict` does the full pipeline — Metal descriptors → CoreML → 4-draw env pooling →
composition decode → conformal set. Pass `decode:`/`conformal:`/`envDraws:` to tune it.
**Never** hand the model masses or any file metadata (see the project's hard rules).

## Core AI (`.aimodel`) — the 2026 backend

[Core AI](https://developer.apple.com/documentation/coreai) is Apple's successor to Core ML
(Xcode 27 / macOS·iOS 27+). Only the **model backend** changes — the Metal descriptor front
end, decode, and conformal code above are backend-agnostic and unchanged.

- **Producing the model:** derive `.aimodel` **from the fp32 checkpoint** (not from ONNX) via
  `coreai-torch`, which is what `scripts/export_model.py --coreai` does — the same
  `ExportModel` wrapper (fixed B=1, T=8, inputs `rdf,pair_extra,frac,glob,mask[,env_d,env_t]`
  → `log_probs`) is handed to `torch.export.export` → `TorchConverter().to_coreai()` →
  `AIProgram.save_asset(...)`. This avoids the int8 quantization of the committed deploy ONNX
  (fp32 `.aimodel` ≈ 4.2 MB). Verified end-to-end on the real architecture (env + no-env),
  export-vs-torch Δ = 0. Conversion runs on macOS 26+; the asset targets OS 27.
- **Running it on-device:** add a `CoreAIRunner` sibling to `CoreMLRunner` that loads the
  `.aimodel` and fills the same seven named inputs. Mirror the Swift runtime in
  [`apple/coreai-models`](https://github.com/apple/coreai-models) once building against the
  Xcode 27 SDK; gate it with `#if canImport(CoreAI)` / `@available(macOS 27, iOS 27, *)` so
  the package still builds on Xcode 26. `TypeSHI` takes any runner exposing `logProbs(_:)`.
- **Bonus:** `coreai-torch` exposes `TorchMetalKernel` (inline Metal kernels in the graph) —
  a future option to fuse `InaccurateRDFCalculator`'s descriptor kernels into the model asset.

## Descriptor contract (keep in sync with `src/typeid2elem/descriptors.py`)

| output | shape | notes |
|---|---|---|
| `rdf` | (T,T,64) | smeared, ideal-gas-normalized `g_ab(r)`, ordered (a←b), clip ≤1e4 |
| `pair_extra` | (T,T,8) | `[log1p(cn@2,3,4,6), nn_median/8, nn_p10/8, peak_pos, log1p(peak_h)]` |
| `frac` | (T,) | stoichiometric fractions |
| `glob` | (3,) | `[log(N/V), has_cell, 1/T]` — **no temperature/metadata, ever** |
| `env_d`,`env_t` | (T,16,16) | per type, 16 sampled atoms × 16 nearest (distance, partner type); −1 pad |

CoreML I/O (from `scripts/export_model.py`): inputs `rdf, pair_extra, frac, glob, mask,
env_d, env_t` (all padded to fixed **T=8**, `mask`/`env_t` are floats), output `log_probs`
(1,T,94). Real types fill the first `nTypes` rows; the rest are masked padding.

## Parity protocol

```bash
swift build --package-path metal
swift run --package-path metal parity-check metal/parity/golden.json
# add a model to also run the CoreML top-1 leg (Release artifact; not committed):
swift run --package-path metal parity-check metal/parity/golden.json --model path/to/env_codfull_sharp30.mlpackage
```

`parity-check` runs four checks against `parity/golden.json`:

| # | check | tolerance | notes |
|---|---|---|---|
| 1 | `frac`, `glob` | 1e-4 / 1e-3 | exact counting math |
| 1 | `pair_extra` | 5e-3 | robust stats; peak/CN near-exact |
| 1 | `rdf` | 5e-2 max\|Δ\| | smearing/normalization; the *Inaccurate* budget lives here |
| 2 | `env_d`/`env_t` vs `env_seed0` | structural | sampled feature; a crystal makes rows equivalent |
| 3 | decode marginals + conformal set | 1e-6 / exact | Swift `CompositionPrior`+RAPS on `golden.log_probs` vs `parity/decode_golden.json` |
| 4 | top-1 per type (`--model`) | must match | the real acceptance test (needs the `.mlpackage`) |

If top-1 and the 90% conformal set match the reference, the port is good even if the raw
RDF differs — that's the whole point of *Inaccurate*. **Current standing:** rdf Δ≈5e-6,
pair_extra Δ≈5e-6, decode marginals Δ≈5e-9, env exact.

## Verification (the whole suite)

The CoreML `.mlpackage` needs a real Apple device and ships in the Release tarball, and
`uv run` can't build the training env on macOS (torch is cu128-pinned). So the model +
eval legs are verified through the committed **int8 ONNX** deploy model with a small
no-torch venv (`numpy scipy ase onnxruntime matscipy`). All scripts live in `parity/`:

```bash
# 1. descriptors + env + decode/conformal parity (Swift-only, no model):
swift run --package-path metal parity-check metal/parity/golden.json

# 2. end-to-end: Metal descriptors -> deploy ONNX -> top-1 vs golden:
swift run --package-path metal parity-check metal/parity/golden.json --dump /tmp/desc.json
python metal/parity/onnx_bridge.py /tmp/desc.json

# 3. Metal path on the committed real eval files (Data/eval_cases/*), vs Python & truth:
python metal/parity/eval_metal_local.py

# 4. open-box (cell-less) + triclinic descriptor paths the eval files don't cover:
python metal/parity/check_edge_cases.py

# 5. lite (ONNX + decode) accuracy on the local eval subset:
python metal/parity/eval_lite_local.py

# 6. per-backend accuracy on Data/eval_cases (swap only the model; needs the Release
#    .mlpackage/.aimodel). ONNX+CoreML from the no-torch venv, aimodel from a coreai env:
python metal/parity/eval_backends_local.py --backend onnx
python metal/parity/eval_backends_local.py --backend coreml  --model <env_codfull_sharp30.mlpackage>
python metal/parity/eval_backends_local.py --backend aimodel --model <env_codfull_sharp30.aimodel>
```

Standing 3-backend result (18 files / 79 type-ids, top-1/3/5): ONNX-int8 47/65/73,
CoreML-fp16 **48/65/72**, Core AI-fp32 **48/65/72** — the on-device backends match the
6-model production ensemble (47/65/75) on real files.

Regenerate goldens/assets when descriptors, the deploy model, or the decode stats change:
`parity/gen_reference.py`, `parity/gen_decode_golden.py`, `tools/gen_decode_assets.py`.
